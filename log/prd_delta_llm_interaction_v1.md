## 二、本次变更背景

当前 knowledgeBook 已具备完整知识沉淀闭环，但在使用体验和知识质量上仍存在 3 个核心问题：
1. 用户输入什么，系统就存什么，缺少重点提取能力。
2. 相似知识会重复进入知识库，缺少语义级重复识别与处理。
3. 机器人主要依赖命令，普通用户的交互门槛较高。

这些问题并不是孤立功能缺失，而是说明现有系统仍缺少一层统一的语义理解与交互编排能力。

---

## 三、本次变更目标

本次变更的目标是：
1. 让自然语言成为主要交互入口，而不是只依赖命令。
2. 让系统在新增知识前先完成重点提取和结构化整理。
3. 让系统在入库前识别相似知识，并优先推荐合并。
4. 让检索结果从“命中列表”升级为“答案 + 证据 + 关联提示”。

## 五、本次新增的产品能力

## 5.1 自然语言理解
用户可以直接表达需求，例如：
- “帮我记一下这个结论”
- “这条和之前那条是不是重复”
- “查一下之前关于 FTS 的结论”

系统负责理解意图，而不是要求用户先记命令格式。

## 5.2 重点提取与结构化整理
新增知识前，系统会先提取：
- 标题
- 一句话摘要
- 关键要点
- 标签
- 推荐分类

保留原文的同时，生成结构化知识内容供确认与展示使用。

## 5.3 相似知识识别
系统新增前需召回相似知识，并至少区分：
- 合并候选
- 补充候选
- 冲突候选
- 可直接新建

## 5.4 冲突处理
若系统识别出相同主题但结论冲突的知识，必须让用户选择：
- 保留新版本
- 保留旧版本
- 两者并存并标记冲突
- 取消入库

## 5.5 展示层升级
检索结果不再只返回命中条目，而是优先提供：
- 面向问题组织后的答案
- 支撑答案的知识条目
- 冲突知识提示
- 关联知识提示
- 飞书文档位置链接

---

## 六、本次变更后的关键产品规则

## 6.1 新知识必须确认后入库
本次升级后，新增知识仍不能由模型直接入正式库。

统一规则：
- 先提取为候选草稿
- 再展示确认内容
- 用户确认后正式入库

## 6.2 相似知识默认推荐合并
对于高度相似知识：
- 默认推荐合并
- 不建议一开始就强制自动合并
- 最终动作由用户确认

## 6.3 冲突必须由用户决策
当系统发现主题相同但结论冲突时，不能自动覆盖，也不能悄悄丢弃某一条知识。

## 6.4 展示答案必须可追溯
系统给出的答案或归纳，必须能回到具体知识条目和文档位置，而不是只给模型生成文本。

---

## 七、本次变更影响的核心流程

## 7.1 新增知识流程变化
原流程：
```plaintext
输入 → 候选 → 确认 → 入库
```

升级后：
```plaintext
输入 → 意图识别 → 重点提取 → 相似知识检测 → 合并/冲突建议 → 用户确认 → 入库
```

## 7.2 查询流程变化
原流程：
```plaintext
提问 → 检索 → 返回结果
```

升级后：
```plaintext
提问 → 意图识别 → 检索 → 聚合证据 → 组织答案 → 返回答案+证据
```

## 7.3 维护流程变化
知识更新、软删除、恢复等流程仍保留，但后续可以继续纳入统一的 LLM 编排入口，以降低用户操作门槛。

---

## 八、对用户体验的主要影响

## 8.1 正向影响
- 更少依赖命令记忆
- 新增知识前能看到更清晰的结构化内容
- 知识库重复率下降
- 检索结果更接近可直接使用的答案

## 8.2 需要接受的变化
- 新增知识前确认步骤更明确
- 冲突知识处理时，用户需要主动选择策略
- LLM 参与后，系统会更“智能”，但也更依赖确认与证据约束

---

## 九、验收重点

本次变更重点验收以下结果：
- 用户可通过自然语言完成主要知识操作
- 候选知识结构化结果可读、可确认
- 相似知识能被识别并给出处理建议
- 冲突知识不会被静默覆盖
- 查询结果能返回答案与证据，而不只是命中列表
- 新增知识仍保持“确认后入库”的安全边界
