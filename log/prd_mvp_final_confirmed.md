## 二、产品概述

### 2.1 产品定位
knowledgeBook 是一款面向个人与轻量团队场景的知识沉淀助手。

它以飞书机器人为主要交互入口，以飞书文档为主要浏览载体，以数据库为唯一事实源，帮助用户完成：
- 信息输入
- 候选知识整理
- 分类沉淀
- 知识检索
- 知识更新维护
- 文档化展示

### 2.2 产品目标
knowledgeBook 要解决的问题不是“把聊天记录存下来”，而是把高频沟通中的有效信息，沉淀为：
- 可确认的
- 可分类的
- 可检索的
- 可维护的
- 可复用的
- 可持续演进的知识资产

### 2.3 产品形态
产品由 3 个核心部分组成：

| 组成部分 | 主要职责 |
| --- | --- |
| 飞书机器人 | 输入、确认、检索、维护、触发操作 |
| 数据库与后端服务 | 事实存储、状态流转、检索、版本、审计、同步 |
| 飞书文档 | 分类化展示、层级浏览、文档位置跳转 |

## 三、用户与使用场景

### 3.1 目标用户
当前优先面向：
- 研发工程师
- 产品经理
- 项目协作者
- 有持续知识沉淀需求的个人知识工作者

### 3.2 典型场景

#### 场景 A：记录新知识
用户把聊天中的结论、方案、注意事项发给机器人，希望系统先整理再入库。

#### 场景 B：整理零散信息
用户给出一段较长文本，希望系统提取重点并形成结构化知识。

#### 场景 C：处理相似或重复知识
用户新增内容时，系统发现已有相似知识，并提示合并、补充或冲突处理。

#### 场景 D：查询历史结论
用户通过自然语言提问，快速找回之前沉淀的方案、规则、决策结论。

#### 场景 E：维护已有知识
用户需要更新知识、改分类、软删除、恢复知识，或把文档手工修改显式同步回系统。

#### 场景 F：文档化浏览
用户希望以分类树和飞书文档结构，长期浏览、复用、沉淀知识。

## 四、产品目标与非目标

## 4.1 核心目标

### 目标 1：降低知识沉淀门槛
用户不需要打开独立后台，即可在飞书中完成知识输入和后续处理。

### 目标 2：提升知识质量
系统不只做原文存储，而要支持摘要提取、分类、版本、去重建议和结构化展示。

### 目标 3：提升知识复用效率
用户能够通过自然语言快速检索知识，并得到可直接复用的答案与文档入口。

### 目标 4：保证知识可控与可维护
知识需要支持确认、更新、软删除、恢复、文档同步和审计回溯。

## 4.2 非目标
当前版本明确不以以下方向为主目标：
- 独立 Web 控制台
- 复杂多人协作与权限系统
- 群聊自动监听与大规模自动采集
- 原始聊天完整追溯回放
- 面向外部平台的开放式分享产品

## 五、完整功能范围

## 5.1 输入能力
支持以下输入方式：
- 用户与机器人私聊输入
- 用户主动转发聊天记录给机器人
- 显式命令输入
- 自然语言输入

输入后，系统需统一进入知识整理链路，而不是直接落正式库。

## 5.2 候选知识生成
系统需要把原始输入整理成候选知识，至少包含：
- 标题
- 一句话摘要
- 关键要点
- 标签
- 推荐分类
- 置信度
- 原始输入内容

## 5.3 用户确认
候选知识必须允许用户执行：
- 确认沉淀
- 忽略
- 稍后处理
- 修改标题 / 内容 / 分类后再确认

## 5.4 分类体系
系统支持多级分类，要求：
- 最大层级暂定 5 级
- 优先采用用户已有分类标准
- 若无用户标准，系统生成初始分类结构并做推荐
- 高置信分类可自动采纳，但结果必须可见且可修改

## 5.5 重复知识识别
系统需要支持相似知识识别，至少区分：
- 建议合并
- 建议作为补充
- 结论冲突
- 可直接新建

对于相似知识，系统默认推荐合并；若存在冲突，必须要求用户做明确选择。

## 5.6 待办能力
被标记为“稍后处理”的候选知识进入待办列表，支持：
- 查看待办
- 继续处理单条待办
- 将待办转为沉淀或忽略

## 5.7 检索能力
用户可以通过自然语言或命令检索知识，系统需返回：
- 可直接使用的答案摘要
- 相关知识条目
- 飞书文档位置链接
- 关联知识或冲突知识提示

## 5.8 知识维护能力
已沉淀知识需支持：
- 更新标题
- 更新内容
- 调整分类
- 软删除
- 恢复
- 合并知识
- 作为补充追加到已有知识

## 5.9 飞书文档展示能力
系统需把知识同步到飞书文档，用于：
- 分类层级浏览
- 子文档导航
- 文档位置跳转
- 结构化展示知识内容

## 5.10 文档修改同步回收
用户直接修改飞书文档后，可通过显式指令要求系统：
- 读取变更内容
- 与当前知识版本对比
- 生成新版本并回写数据库

默认情况下，飞书文档手工修改不自动回写数据库。

## 5.11 部署与运维能力
产品需支持：
- 本地或服务器部署
- Docker Compose 启动
- 健康检查
- 基本诊断命令
- 用户部署手册

## 六、关键产品规则

## 6.1 数据事实源规则
- 数据库是唯一事实源
- 飞书文档是派生展示层
- 文档内容变化不能默认反向主导数据库

## 6.2 入库规则
- 新增知识先进入候选或草稿状态
- 用户确认后才成为正式知识
- 不能由模型直接跳过确认写入正式库

## 6.3 相似知识规则
- 系统新增前需进行相似知识检测
- 相似时默认推荐合并
- 冲突时必须用户显式选择处理方式

## 6.4 展示规则
系统展示给用户的内容至少分为三类：
- 原始输入
- 结构化知识
- 面向问题组织后的答案或说明

## 6.5 文档手改规则
- 默认不自动回收飞书文档手改内容
- 仅在用户显式发起同步时回收
- 未同步的手改内容，后续可能被数据库同步覆盖

## 七、核心用户流程

## 7.1 新增知识主流程
```plaintext
用户输入内容
→ 系统生成候选知识
→ 系统推荐分类并检测相似知识
→ 用户确认沉淀 / 忽略 / 稍后处理
→ 若确认则写入正式知识与版本
→ 同步到飞书文档
```

## 7.2 重复知识处理流程
```plaintext
用户新增内容
→ 系统检测相似知识
→ 返回合并 / 补充 / 冲突建议
→ 用户选择处理方式
→ 系统执行新建、合并、补充或冲突保留
```

## 7.3 查询知识流程
```plaintext
用户发起问题
→ 系统检索候选知识
→ 组织答案摘要
→ 返回答案 + 相关知识 + 文档位置链接
```

## 7.4 知识维护流程
```plaintext
用户选择已沉淀知识
→ 更新内容 / 调整分类 / 软删除 / 恢复
→ 系统生成新版本并同步文档
```

## 7.5 文档回收流程
```plaintext
用户手动修改飞书文档
→ 用户显式发起同步指令
→ 系统读取修改部分
→ 与数据库比对后生成新版本
→ 重新同步文档
```

## 八、交互要求

## 8.1 主交互入口
主交互入口为飞书机器人。

用户应能通过：
- 自然语言
- 显式命令
- 卡片操作

完成主要流程。

## 8.2 机器人消息要求
机器人反馈需要尽量满足：
- 一轮消息可完成一个小决策
- 关键动作均有反馈
- 复杂状态有明确说明
- 尽量减少额外学习成本

## 8.3 飞书文档要求
飞书文档应主要承担：
- 浏览
- 分类化沉淀
- 结构化展示
- 稳定链接入口

而不是默认编辑主入口。

---

## 九、检索与展示要求

## 9.1 检索结果结构
标准检索结果至少包含：
- 答案摘要
- 知识标题
- 分类路径
- 更新时间
- 飞书文档位置链接

## 9.2 展示层要求
系统面向用户展示知识时，需要支持：
- 单条知识摘要展示
- 多条知识归纳展示
- 冲突知识差异说明
- 与当前问题强相关的知识证据展示

## 十、知识生命周期

## 10.1 候选知识生命周期
- 已提取
- 待确认
- 已忽略
- 稍后处理
- 已确认

## 10.2 正式知识生命周期
- 生效中
- 已更新
- 已合并进其他知识
- 已软删除
- 冲突标记中

## 10.3 删除规则
- 软删除后默认不参与检索
- 数据保留 30 天
- 30 天内支持恢复
- 过期后可永久清理

---

## 十一、版本与审计要求

系统需要保留：
- 正式知识版本历史
- 关键操作日志
- 同步任务状态
- 合并关系与冲突处理结果

以保证后续可维护、可追溯、可恢复。

---

## 十二、成功标准

## 12.1 功能完成标准
- 支持输入、候选、确认、分类、检索、更新、软删除、恢复、同步回收
- 支持飞书文档分层展示与位置跳转
- 支持相似知识识别与处理建议

## 12.2 体验完成标准
- 用户不依赖独立后台也能完成主流程
- 机器人交互足够清晰，关键动作无歧义
- 检索结果可直接复用，而不是仅返回原始记录

## 12.3 系统完成标准
- 数据库始终是唯一事实源
- 文档同步失败可补偿与重试
- 所有关键状态有记录可追踪
