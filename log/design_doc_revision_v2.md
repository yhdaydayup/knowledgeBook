## 二、设计目标

### 2.1 设计定位
小帮手 MVP 的设计目标不是构建一个完整的知识后台，而是围绕**飞书机器人交互**建立一套低打扰、低学习成本、强可控的知识沉淀体验。

### 2.2 设计目标
- 让用户通过飞书聊天即可完成知识输入、确认、检索和维护
- 让分类推荐与人工确认形成协同，而不是额外增加负担
- 让知识通过飞书文档按层级结构沉淀，便于长期浏览和复用
- 让用户能够方便地更新、调整分类或软删除知识
- 让用户能根据部署手册快速接入和使用系统

### 2.3 MVP 设计边界
本版本明确不包含：
- 独立 Web 页面
- 仪表盘 / 后台页面
- 原始聊天追溯页
- 分享页 / MCP 授权页
- 多人协作页面

## 三、设计原则

### 3.1 品牌设计关键词
- **可信**：用户能理解系统当前状态和结果
- **克制**：不做过度打扰，不堆叠复杂操作
- **智能**：通过 AI 提供推荐，但不替用户做最终决策
- **高效**：在对话中直接完成操作
- **温和**：反馈清晰但不过度警示

### 3.2 交互原则
- **消息流优先**：以飞书对话消息代替页面流转
- **单轮可决策**：一条卡片尽量支持一个完整小决策
- **系统先推荐，用户再确认**：分类、沉淀、更新均遵循该原则
- **所有关键动作均有反馈**：沉淀、忽略、稍后处理、更新、软删除都要给出明确结果
- **支持恢复和继续处理**：稍后处理进入待办；软删除需二次确认
- **分类标准优先**：若用户已有分类标准，系统交互优先围绕用户标准展开；若没有，再使用智能推荐

### 3.3 文案原则
按钮与操作文案统一采用动词形式：
- 沉淀
- 忽略
- 稍后处理
- 使用推荐分类
- 修改分类
- 更新知识
- 软删除知识
- 查看文档

---

## 四、交互载体设计

### 4.1 飞书机器人
飞书机器人承担以下核心交互：
- 输入知识内容
- 接收候选知识卡片
- 确认沉淀 / 忽略 / 稍后处理
- 接受或修改分类
- 查看待办
- 检索知识
- 更新知识
- 软删除知识

### 4.2 飞书文档
飞书文档承担以下展示职责：
- 按分类层级展示已沉淀知识
- 提供知识浏览入口
- 提供文档内位置跳转
- 允许上层文档嵌套下层文档链接

### 4.3 载体协作原则
- 机器人负责“操作”
- 文档负责“展示”
- 机器人给用户返回文档位置链接
- 文档不承载默认编辑主入口
- 用户若手动修改文档，需通过显式同步指令让小帮手回收修改
- 若未发出同步指令，后续数据库同步会覆盖手工修改

## 五、机器人消息结构规范

### 5.1 通用结构
每类机器人消息应尽量包含：
1. 标题
2. 摘要
3. 核心信息字段
4. 推荐动作按钮
5. 次级操作入口
6. 状态反馈区

### 5.2 常用消息类型
- 候选知识卡片
- 分类推荐卡片
- 待办列表卡片
- 检索结果卡片
- 知识维护卡片
- 文档导航卡片
- 异常提示卡片

### 5.3 状态表现
消息至少支持以下状态表达：
- 加载中
- 处理成功
- 处理失败
- 无结果
- 低置信度提醒
- 自动分类已执行提示
- 部分完成

## 六、核心流程设计

## 6.1 输入与候选生成流程

### 6.1.1 场景
- 用户直接给小帮手发消息
- 用户转发聊天记录给小帮手

### 6.1.2 系统反馈流程
```plaintext
收到输入 → 提示“已接收，正在整理” → 生成候选知识卡片 → 返回候选结果
```

### 6.1.3 候选知识卡片字段
- 标题
- 摘要
- 内容预览
- 标签
- 推荐分类
- 置信度

### 6.1.4 候选卡片主动作
- 沉淀
- 忽略
- 稍后处理

### 6.1.5 候选卡片次动作
- 修改标题
- 修改内容
- 修改分类

---

## 6.2 分类推荐与指定流程

### 6.2.1 交互目标
减少用户从零选择分类的成本，同时保留最终决策权。

### 6.2.2 分类来源优先级
分类交互优先级如下：
1. 用户预设或明确提供的分类标准
2. 系统已有分类树
3. AI 智能推荐的初始一级、二级分类结构

### 6.2.3 交互方式
系统返回：
- 推荐分类路径
- 可选备选分类
- 手动指定分类入口
- 必要时的新建分类入口

### 6.2.4 自动分类规则
- 高置信分类允许自动归类
- 自动归类后仍需在反馈中明确告知用户结果
- 用户可一键修改自动归类结果
- 对于不满足阈值的分类，仍采用人工确认

### 6.2.5 分类路径展示规则
分类使用可读路径展示，例如：

```plaintext
工作 / 项目A / 接口设计 / 登录流程
```

### 6.2.6 多级分类设计要求
- 最大层级暂定 5 级
- 路径必须可复核
- 路径一旦确认，应在后续检索和文档中保持一致展示

---

## 6.3 稍后处理待办流程

### 6.3.1 定义
所有被标记为“稍后处理”的候选知识进入独立待办列表。

### 6.3.2 待办能力
- 查看待办列表
- 继续处理单条待办
- 批量处理待办
- 将待办转为沉淀或忽略

### 6.3.3 待办列表卡片内容
- 待办数量
- 每条候选摘要
- 推荐分类
- 主动作：继续处理 / 批量沉淀 / 批量忽略

### 6.3.4 状态设计
- 无待办
- 待办较多时分页展示
- 待办失效时提示重新整理

---

## 6.4 检索流程

### 6.4.1 检索入口
用户直接通过飞书机器人自然语言提问。

### 6.4.2 检索返回结构
- AI 总结答案
- 相关知识条目
- 飞书文档对应位置链接

### 6.4.3 检索结果卡片字段
- 答案摘要
- 相关知识标题
- 分类路径
- 更新时间
- 查看文档按钮

### 6.4.4 明确不展示内容
MVP 检索结果不展示：
- 原始聊天片段
- 原始消息链接
- 会话追溯链路

### 6.4.5 检索后的继续动作
- 重新提问
- 查看文档
- 对知识发起更新
- 对知识发起软删除

---

## 6.5 知识更新流程

### 6.5.1 支持更新项
- 标题
- 内容
- 分类

### 6.5.2 交互流程
```plaintext
用户发起更新 → 机器人定位知识 → 展示当前摘要 → 用户给出修改内容 → 返回确认卡片 → 用户确认 → 同步飞书文档
```

### 6.5.3 更新反馈
- 已更新知识
- 已调整分类
- 正在同步飞书文档
- 同步成功 / 同步失败

## 6.5.4 文档手动修改同步回收
当用户直接在飞书文档中手动修改知识后，可通过显式指令要求小帮手读取修改部分并同步回数据库。

#### 交互规则
- 默认不自动同步文档手动修改
- 用户需发出明确同步指令
- 若未发出同步指令，后续数据库同步文档时将直接覆盖手工修改

#### 反馈要求
- 明确提示“已读取文档修改并同步回数据库”
- 若无法识别修改范围，提示用户补充知识标题、分类路径或文档位置
- 同步成功后提示已生成新版本

---

## 6.6 知识软删除流程

### 6.6.1 交互目标
保证知识库可持续清理，同时避免误删，并让用户清楚理解 30 天恢复窗口。

### 6.6.2 流程
```plaintext
用户发起软删除 → 展示待删除知识摘要 → 二次确认 → 执行软删除 → 返回结果反馈 → 提示 30 天恢复入口
```

### 6.6.3 反馈要求
- 明确说明该知识默认不再参与检索
- 说明该操作为软删除，不是物理删除
- 明确提示“该知识将在 30 天后永久清理，可在 30 天内恢复”
- 提供“恢复知识”快捷入口或明确恢复指令
- 若文档同步未完成，需提示“文档更新稍后完成”

## 七、分类体系设计规范

### 7.1 分类结构建议
建议采用 2~5 级分类：
- 一级：领域
- 二级：子领域 / 模块
- 三级：主题 / 场景
- 四级：具体问题类型
- 五级：精细知识点（可选）

### 7.2 分类交互要求
- 用户能理解当前知识属于哪条路径
- 用户能快速修改路径
- 路径名称清晰，不使用难懂缩写
- 系统优先复用已有分类，不鼓励过度分叉

### 7.3 分类展示位置
分类路径应出现在：
- 候选知识卡片
- 待办卡片
- 检索结果卡片
- 飞书文档层级中

---

## 八、飞书文档展示规范

### 8.1 展示原则
- 飞书文档仅负责展示，不负责编辑主流程
- 文档结构必须与分类结构基本一致
- 不能把所有知识平铺进单篇长文档
- 以分类子文档为主，避免文档膨胀

### 8.2 建议结构
```plaintext
知识库总览文档
├── 一级分类子文档
│   ├── 二级分类子文档 / 二级标题
│   │   ├── 三级分类子文档 / 三级标题
│   │   │   ├── 四级/五级标题
│   │   │   │   ├── 具体知识条目
```

### 8.3 层级组织策略
- 优先采用子文档嵌套
- 在子文档内部使用多级标题承接细粒度层级
- 当内容过多时，自动拆成下一级子文档
- 当内容过少时，自动合并为当前文档内标题层级

### 8.4 上层文档职责
- 展示分类总览
- 展示子分类入口
- 嵌套下层文档链接

### 8.5 下层文档职责
- 承载具体知识条目
- 展示知识正文与更新时间
- 提供稳定位置链接

### 8.6 文档内知识条目模板
每条知识建议包含：
- 标题
- 分类路径
- 摘要
- 正文
- 标签
- 更新时间

### 8.7 链接规范
- 检索结果返回文档位置链接
- 用户点击后直接落到知识所在位置
- 不返回原始聊天链接

## 九、异常与边界场景设计

### 9.1 空状态
- 无候选知识
- 无待办
- 无检索结果
- 无对应分类文档

### 9.2 错误状态
- 飞书消息发送失败
- 分类写入失败
- 文档同步失败
- 检索超时
- 未找到要更新的知识

### 9.3 风险控制
- 软删除需二次确认
- 低置信度知识不默认沉淀
- 批量操作后必须返回结果摘要
- 自动分类需可一键修正

## 十一、MVP 设计验收标准

- [ ] 用户可通过飞书机器人完成知识输入
- [ ] 候选知识卡片可完成沉淀 / 忽略 / 稍后处理
- [ ] 系统可推荐分类，用户可修改或指定分类
- [ ] 高置信分类可自动归类且可修正
- [ ] 稍后处理内容可进入独立待办
- [ ] 检索结果默认返回“答案 + 条目 + 文档位置链接”
- [ ] 用户可通过机器人发起知识更新与软删除
- [ ] 用户手动修改文档后，可通过显式指令触发同步回收
- [ ] 未触发同步指令时，界面文案明确告知“手工修改可能被覆盖”
- [ ] 飞书文档可按最多 5 级层级结构组织知识
- [ ] 检索结果可稳定跳转到对应文档位置
- [ ] 设计交付中包含部署接入说明所需示意内容
