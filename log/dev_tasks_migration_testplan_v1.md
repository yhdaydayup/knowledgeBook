## 二、第一阶段开发目标

第一阶段的目标不是一次性完成全部 MVP，而是先打通一条**最小可运行主链路**：

> 飞书机器人输入 → 候选知识生成 → 人工确认 → 分类入库 → 文档同步 → 检索返回文档位置链接

在此基础上，再补齐：
- 稍后处理待办
- 知识更新
- 软删除与恢复
- 文档修改同步回收
- 部署与联调

### 第一阶段建议拆分为 4 个里程碑

| 里程碑 | 目标 |
| --- | --- |
| M1 | 基础框架、数据库、飞书接入打通 |
| M2 | 候选知识、分类推荐、确认入库打通 |
| M3 | 文档同步、检索、知识维护打通 |
| M4 | 软删除恢复、文档回收、部署验收闭环 |

## 四、数据库 Migration 顺序

## 4.1 拆分原则
- 先建基础表，再建业务表，再建同步/日志表，最后补索引
- 每一步 migration 保持可单独执行、可单独回滚
- 避免一个 migration 混入过多业务变更

---

## 4.2 推荐 Migration 拆分

### M001_users.sql
#### 内容
- 创建 `users`

#### 目标
- 建立用户主表

#### 回滚注意
- 无外键依赖，可直接回滚

---

### M002_categories.sql
#### 内容
- 创建 `categories`

#### 目标
- 建立分类树基础能力

#### 回滚注意
- 若已有知识挂载分类，则不可直接删表

---

### M003_knowledge_drafts.sql
#### 内容
- 创建 `knowledge_drafts`
- 创建 draft 相关索引

#### 目标
- 打通候选知识链路

#### 回滚注意
- 回滚前需确认没有依赖 draft 的知识生成流程在运行

---

### M004_knowledge_items.sql
#### 内容
- 创建 `knowledge_items`
- 建立 `draft_id`、`primary_category_id` 外键

#### 目标
- 建立正式知识主表

#### 回滚注意
- 与后续版本、同步映射有关，不建议单独在生产回滚

---

### M005_knowledge_versions.sql
#### 内容
- 创建 `knowledge_versions`
- 建立唯一索引 `(knowledge_id, version_no)`

#### 目标
- 建立知识版本管理能力

#### 回滚注意
- 回滚前需确认版本数据已处理

---

### M006_ai_category_recommendations.sql
#### 内容
- 创建 `ai_category_recommendations`

#### 目标
- 记录 AI 分类推荐结果

#### 回滚注意
- 仅影响推荐分析，不影响主数据存活

---

### M007_doc_sync_mappings.sql
#### 内容
- 创建 `doc_sync_mappings`

#### 目标
- 建立知识与飞书文档位置映射

#### 回滚注意
- 回滚后文档位置跳转能力失效

---

### M008_sync_tasks.sql
#### 内容
- 创建 `sync_tasks`

#### 目标
- 建立异步任务与补偿机制

#### 回滚注意
- 回滚前需清空未执行任务或迁移任务数据

---

### M009_operation_logs.sql
#### 内容
- 创建 `operation_logs`

#### 目标
- 建立审计与操作记录能力

#### 回滚注意
- 回滚会损失审计记录

---

### M010_indexes.sql
#### 内容
- 添加所有业务索引

#### 目标
- 提升检索、任务扫描、分类查询效率

#### 回滚注意
- 可单独回滚索引，不影响表结构

---

### M011_seed_categories.sql
#### 内容
- 在无用户分类标准时，写入默认一级 / 二级分类种子

#### 目标
- 提供智能分类初始落点

#### 回滚注意
- 仅能删除未被使用的种子分类

---

## 4.3 建议执行顺序
```plaintext
M001 → M002 → M003 → M004 → M005 → M006 → M007 → M008 → M009 → M010 → M011
```

## 4.4 生产变更注意事项
- 所有 migration 必须先在测试环境演练
- 大表索引建议避开业务高峰执行
- 如需回滚，优先回滚应用，再回滚 schema

## 五、测试计划与验收清单

## 5.1 测试目标
确保 MVP 核心链路可运行、可恢复、可同步、可部署。

## 5.2 测试层次

### 单元测试
覆盖：
- 命令解析
- 分类路径计算
- 高置信自动分类判定
- 软删除恢复窗口判断
- 错误码映射

### 集成测试
覆盖：
- 数据库 CRUD
- 版本写入
- 同步任务入队与执行
- 文档映射读写
- `/kb sync-from-doc` 回收链路

### 接口测试
覆盖：
- `/api/v1/knowledge`
- `/api/v1/knowledge/search`
- `/api/v1/knowledge/{id}`
- `/api/v1/knowledge/{id}/restore`
- `/api/v1/knowledge/{id}/sync-from-doc`
- `/api/v1/categories/tree`

### 端到端测试
覆盖：
- 飞书输入 → 候选 → 确认 → 文档同步 → 检索
- 更新 → 文档再同步
- 软删除 → 恢复
- 文档手改 → 显式回收

## 5.3 核心测试场景

### 场景 1：候选确认主链路
```plaintext
输入消息 → 生成候选 → 确认沉淀 → 分类入库 → 文档同步 → 检索返回链接
```
验收：
- [ ] draft 状态流转正确
- [ ] knowledge 创建成功
- [ ] 文档位置链接可打开

### 场景 2：高置信自动分类
验收：
- [ ] 满足阈值时自动分类
- [ ] 自动分类结果可被人工改掉

### 场景 3：稍后处理待办
验收：
- [ ] later 状态可见
- [ ] later 可转 approve / ignore

### 场景 4：知识更新
验收：
- [ ] 生成新版本
- [ ] 文档同步到新内容

### 场景 5：软删除与恢复
验收：
- [ ] 软删除后默认检索不到
- [ ] 30 天内可恢复
- [ ] 恢复后文档链接仍有效

### 场景 6：超期清理
验收：
- [ ] 超过 `purge_at` 后被清理
- [ ] 清理后不可恢复

### 场景 7：文档修改同步回收
验收：
- [ ] 手动修改文档但不发指令时，不回写数据库
- [ ] 发出 `/kb sync-from-doc` 后生成新版本
- [ ] 无差异时返回 `DOC_BACKFILL_NO_DIFF`

### 场景 8：部署验证
验收：
- [ ] `docker compose up -d` 可成功启动
- [ ] `healthz` / `readyz` 正常
- [ ] 飞书事件能回调
